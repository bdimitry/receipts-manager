# Дорожная карта проекта

## Текущий статус

На текущий момент в проекте закрыты:

- Шаг 1: технический каркас backend-приложения
- Шаг 2: foundation для `User + JWT auth`

Уже реализовано:

- Spring Boot backend
- PostgreSQL + Flyway
- LocalStack для `S3` и `SQS`
- health endpoint
- регистрация и логин пользователя
- JWT access token
- защищенные endpoints
- интеграционные тесты на `TestRestTemplate`
- auth integration tests через Testcontainers PostgreSQL

## Почему следующий шаг именно Purchase

После того как в проекте появился `User + JWT`, самым логичным следующим шагом становится первая бизнес-сущность, привязанная к пользователю.

Для этого лучше всего подходит `Purchase`, потому что:

- это первая реальная пользовательская запись в системе
- на `Purchase` потом будут опираться чеки, отчеты и аналитика
- этот шаг позволяет проверить полный вертикальный срез:
  - миграция БД
  - JPA-сущность
  - repository
  - service
  - controller
  - авторизация доступа к данным
  - интеграционные тесты

## Следующие этапы

### Шаг 3

`Purchase CRUD`

Что должно появиться:

- таблица `purchases`
- CRUD для покупок
- привязка покупки к текущему пользователю
- фильтрация списка покупок
- интеграционные тесты на пользовательский доступ

### Шаг 4

`Receipt metadata + upload foundation`

Что должно появиться:

- сущность `Receipt`
- хранение метаданных чека в БД
- загрузка файла в `S3`
- привязка чека к пользователю и, опционально, к покупке

### Шаг 5

`ReportJob + SQS foundation`

Что должно появиться:

- сущность `ReportJob`
- создание задачи на генерацию отчета
- отправка сообщения в `SQS`
- статусы `NEW / PROCESSING / DONE / FAILED`

### Шаг 6

`Report generation`

Что должно появиться:

- генерация `CSV`
- сохранение отчета в `S3`
- получение ссылки на скачивание
- уведомление о готовности отчета

## Принцип дальнейшей разработки

Дальше лучше двигаться не "по слоям", а "по вертикальным срезам":

- сначала одна завершенная пользовательская функция
- потом следующая

То есть:

1. сущность
2. миграция
3. repository
4. service
5. controller
6. security/access rules
7. integration tests

Так проект будет оставаться в рабочем состоянии после каждого этапа.

## Дополнительно зафиксировано в плане

Следующими крупными блоками после завершения текущих OCR/UI/currency задач также остаются:

- `Admin panel`
- `Google authentication`
- `Kubernetes / production-style deployment`

### Admin panel

Нужно предусмотреть отдельную административную зону для:

- просмотра пользователей
- просмотра report jobs
- просмотра OCR статусов чеков
- диагностики ошибок интеграций и уведомлений

### Google authentication

Нужно добавить вход через Google как дополнительный способ аутентификации рядом с текущим `JWT + email/password` flow.

### Kubernetes

Kubernetes остается обязательной частью дальнейшего плана:

- manifests или Helm chart
- локальная проверка через `kind` или `k3d`
- production-style deployment artifacts
