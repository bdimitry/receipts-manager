# Шаг 3: Purchase CRUD

## Цель шага

Реализовать первую бизнес-сущность проекта — `Purchase` — и дать пользователю возможность создавать, просматривать и удалять свои покупки через защищенный REST API.

Этот шаг должен превратить проект из "auth foundation" в "первую рабочую пользовательскую функцию".

## Почему это следующий шаг

Сейчас в проекте уже есть:

- `User`
- JWT-аутентификация
- механизм получения текущего пользователя
- интеграционные тесты через `TestRestTemplate`

Значит можно безопасно переходить к данным, которые принадлежат конкретному пользователю.

`Purchase` — лучшая первая бизнес-сущность, потому что:

- она простая
- она нужна для будущих отчетов
- на ней удобно отладить user-scoped доступ

## Что входит в шаг 3

### Новая сущность

`Purchase`

Минимальные поля:

- `id`
- `userId`
- `title`
- `category`
- `amount`
- `purchaseDate`
- `storeName`
- `comment`
- `createdAt`

Рекомендации:

- `amount` хранить как `BigDecimal`
- `purchaseDate` хранить как `LocalDate`
- `storeName` и `comment` можно сделать nullable
- `title` и `category` обязательны

### База данных

Добавить миграцию:

- `V3__create_purchases_table.sql`

В таблице должны быть:

- первичный ключ
- внешний ключ на `users`
- индексы минимум по `user_id` и `purchase_date`

### REST API

Добавить endpoints:

- `POST /api/purchases`
- `GET /api/purchases`
- `GET /api/purchases/{id}`
- `DELETE /api/purchases/{id}`

Рекомендуемые query params для списка:

- `year`
- `month`
- `category`

### Access control

Покупки должны быть привязаны к текущему пользователю из JWT.

Правила:

- пользователь видит только свои покупки
- пользователь не может получить чужую покупку
- пользователь не может удалить чужую покупку

Для чужих данных лучше возвращать `404`, а не `403`, чтобы не светить существование чужих идентификаторов.

### DTO

Минимальный набор:

- `CreatePurchaseRequest`
- `PurchaseResponse`

Опционально:

- `PurchaseListItemResponse`

### Сервисный слой

Нужны операции:

- создать покупку
- получить список покупок текущего пользователя
- получить покупку по id
- удалить покупку

## Что не входит в шаг 3

- `Receipt`
- загрузка файлов в `S3`
- `ReportJob`
- `SQS`
- генерация отчетов
- уведомления
- редактирование покупки

Если останется время, `PUT/PATCH` можно сделать позже отдельным мини-шагом.

## Критерии готовности

Шаг считается завершенным, если:

- есть таблица `purchases`
- пользователь может создать покупку
- пользователь может получить список только своих покупок
- пользователь может получить одну свою покупку
- пользователь может удалить свою покупку
- без токена purchase endpoints возвращают `401`
- чужие покупки недоступны
- интеграционные тесты проходят

## Обязательные тесты

- создание покупки с валидным токеном
- создание покупки без токена -> `401`
- валидация обязательных полей -> `400`
- список покупок возвращает только данные текущего пользователя
- получение своей покупки -> `200`
- получение чужой покупки -> `404`
- удаление своей покупки -> `204`
